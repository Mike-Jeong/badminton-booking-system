# 배드민턴 예약 관리 시스템 — 요구사항 v2 (개정본)

원본 요구사항(23개 항목) 검토 후 발견된 모호한 지점을 사용자와 확정한 결과를 반영한 개정본.
원본과 충돌하는 항목은 이 문서가 우선한다.

---

## 1. 이번 개정에서 확정된 사항

### 1.1 취소 후 재예약 정책 (원본 6번, 18번 보완)

- 사용자는 특정 예약일에 대해 `CANCELLED` 이력이 있어도 **본인이 직접 다시 예약할 수 있다.**
- "취소 이력이 있으면 자동 재예약하지 않고 관리자가 처리한다"는 규칙은 **`MONTHLY_MEMBER_AUTO`(월 멤버 자동배정) 소스에만 적용**된다.
- 즉 `createBooking`의 중복 체크 로직:
  - 같은 `bookingDayId + normalizedName + normalizedPhone` 조합에 `WAITING` 또는 `CONFIRMED`가 있으면 → 항상 거부 (중복 예약)
  - 같은 조합에 `CANCELLED`만 있고 신규 요청 `source`가 `USER` 또는 `ADMIN` → 새 행(row)으로 생성 허용
  - 같은 조합에 `CANCELLED`만 있고 신규 요청 `source`가 `MONTHLY_MEMBER_AUTO` → 생성하지 않고 스킵 (관리자가 필요 시 수동 처리)

### 1.2 예약일(BookingDay) 세션 정책 (원본 3번, 17번 보완)

- `date`에 unique 제약을 두지 않는다. **같은 날짜에 여러 개의 BookingDay(세션)를 허용**한다.
- 다만 UI에서 세션을 구분할 수 있도록 `BookingDay`에 선택 필드 `label`(예: "오전", "오후", "1부") 추가를 제안한다. 필수 아님, 비어 있으면 시간 미표기.

### 1.3 예약 취소 API 계약 (원본 14번 보완)

- 취소는 `bookingId + phone`을 받아서, 해당 `bookingId`의 `normalizedPhone`과 입력한 전화번호(정규화 후)가 일치할 때만 취소 처리한다.
- 사용자가 `bookingId`를 알아야 하므로, 취소 전 단계로 "전화번호로 내 예약 조회" 엔드포인트를 둔다:
  - 입력: 전화번호
  - 출력: 해당 번호로 된 예약 목록 (bookingId, 날짜, 이름, 상태) — `CANCELLED` 제외 또는 표시만
  - 사용자가 목록에서 선택 → 선택한 `bookingId + phone`으로 취소 요청

### 1.4 공개 예약 페이지 노출 정보 (원본 7번 보완)

- 같은 예약일의 다른 예약자 **이름은 전체 공개**한다 (동호회 특성상 마스킹 불필요).
- 전화번호는 비공개 (이름만 노출, 상태는 확정/대기 여부 정도만 표시 권장).

## 2. 기본값으로 확정한 항목 (원본에 명시 없던 부분)

별도 이견 없으면 아래 기본값으로 진행한다.

- `BookingDay.dayOfWeek`는 사용자 입력이 아니라 `date`로부터 서버가 자동 계산한다.
- 정규화 규칙: 이름은 trim, 전화번호는 숫자만 남김(하이픈·공백 제거).
- 타임존은 Pacific/Auckland(뉴질랜드) 고정 (날짜/요일 계산, "오늘" 대시보드 기준). 해당 타임존은 서머타임(NZDT/NZST)을 사용하므로 고정 offset이 아닌 IANA 타임존 이름으로 처리해야 함.
- `SEPARATED` 모드는 저장 시 `totalSlots = annualSlots + casualSlots` 검증. `COMBINED` 모드에서는 annual/casualSlots 미사용(0).
- 분리 슬롯에서 연 ↔ 캐주얼 간 슬롯 교차 승격은 없음 (대기자 승격은 각자 풀 안에서만 FIFO).
- 월 멤버 자동배정 예약(`MONTHLY_MEMBER_AUTO`)은 항상 `memberType = ANNUAL`로 처리하며, 분리 모드에서는 연 멤버 슬롯을 소비한다.
- 월 멤버 자동배정 조건: `MonthlyMember.isActive`뿐 아니라 연결된 `AnnualMember.isActive`도 함께 확인 (비활성 연 멤버는 자동배정 제외).
- 관리자 세션은 DB 세션 테이블 없이 서명된(signed) HttpOnly 쿠키로 처리 (예: 만료 24시간).
- 관리자가 `WAITING`을 개별 승인할 때는 슬롯 여유만 확인하고 FIFO 순서를 강제하지 않는다 (자동 승격 로직만 FIFO 적용).
- 연 멤버 등록 시 `normalizedName + normalizedPhone` 조합에 유니크 제약을 둔다 (중복 등록 방지).

## 3. 데이터 모델 변경 요약

```ts
BookingDay {
  id
  date
  dayOfWeek      // 서버 계산, 입력 불가
  label          // NEW, optional — 같은 날짜 다중 세션 구분용
  location
  dutyPerson
  totalSlots
  annualSlots
  casualSlots
  slotMode       // SEPARATED | COMBINED
  isOpen
  createdAt
  updatedAt
  // date에 unique 제약 없음 (다중 세션 허용)
}
```

`Booking`, `AnnualMember`, `MonthlyMember` 모델은 원본 그대로 유지.

## 4. 주요 서비스 함수 로직 변경

### createBooking(bookingDayId, name, phone, source)

```
1. 이름/전화번호 정규화
2. 기존 예약 조회 (bookingDayId + normalizedName + normalizedPhone)
3. WAITING/CONFIRMED 존재 → reject (중복)
4. CANCELLED만 존재:
   - source === MONTHLY_MEMBER_AUTO → skip, 생성 안 함
   - source === USER | ADMIN → 새 행 생성 진행
5. determineMemberType(name, phone) → ANNUAL | CASUAL
6. 슬롯 정책 확인 (SEPARATED/COMBINED, memberType 반영)
7. 여유 있으면 CONFIRMED, 없으면 WAITING
8. DB Transaction 내 처리
```

### cancelBooking(bookingId, phone)

```
1. bookingId로 예약 조회
2. 입력 전화번호 정규화 후 booking.normalizedPhone과 일치 확인 (불일치 시 reject)
3. status를 CANCELLED로 변경 (cancelledAt 기록, 레코드 삭제하지 않음)
4. 취소된 예약이 CONFIRMED였다면 promoteWaitingBookings(bookingDayId) 호출
5. DB Transaction 내 처리
```

### lookupBookingsByPhone(phone) — NEW

```
1. 전화번호 정규화
2. normalizedPhone 일치하는 예약 목록 조회 (bookingId, date, name, status)
3. 취소 대상 선택을 위해 사용자에게 반환
```
