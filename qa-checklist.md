# 배드민턴 예약 관리 시스템 — QA 체크리스트

기준 문서: `기획서.md`(에픽별 완료 기준), `requirements.md`
목적: 에픽 단위로 흩어져 있던 완료 기준(Acceptance Criteria)을 기능 단위로 재편집하고 중복을 제거한 뒤, 비기능 테스트와 배포 전 점검 항목을 추가한 통합 QA 체크리스트.

---

## 1. 관리자 인증

- [ ] 올바른 비밀번호 입력 시 로그인에 성공하고 관리자 페이지 전체에 접근 가능하다.
- [ ] 틀린 비밀번호 입력 시 로그인에 실패하고 명확한 오류 메시지가 노출된다.
- [ ] `ADMIN_PASSWORD`가 Vercel 환경변수에 설정되어 있고, 비교 로직이 `crypto.timingSafeEqual`을 사용해 타이밍 공격을 방지한다.
- [ ] 로그인하지 않은 상태로 관리자 페이지 URL에 직접 접근하면 로그인 페이지로 리다이렉트된다.
- [ ] 세션 발급 후 24시간이 지나면 세션이 만료되어 재로그인이 요구된다.
- [ ] 로그아웃 시 세션이 즉시 무효화된다.
- [ ] 일반 사용자 화면에는 관리자 로그인 여부가 노출되지 않는다.

## 2. 예약일 관리

- [ ] 같은 날짜로 여러 개의 예약일(세션)을 라벨을 다르게 하여 생성할 수 있다.
- [ ] 요일 입력 필드가 화면에 없고, 날짜 선택 시 요일이 자동으로 표시된다.
- [ ] 분리 슬롯 모드에서 연 슬롯 + 캐주얼 슬롯 합이 전체 슬롯과 다르면 저장이 실패하고 안내 메시지가 뜬다.
- [ ] 통합 슬롯 모드에서는 연/캐주얼 슬롯 입력 없이 전체 슬롯만으로 생성 가능하다.
- [ ] `isOpen=false`인 예약일은 사용자 공개 화면에 노출되지 않으며, 신청 자체도 불가능하다.
- [ ] 관리자는 예약일을 삭제할 수 있고, 삭제 전 연결된 확정/대기(CONFIRMED/WAITING) 예약 존재 여부를 확인할 수 있는 안내를 받는다. CANCELLED 예약만 있거나 예약이 전혀 없는 예약일은 삭제할 수 있으며, 그 CANCELLED 기록도 예약일과 함께 삭제된다(decisions.md D-17).
- [ ] 슬롯 수(분리 모드면 연/캐주얼 각각)를 현재 확정 인원보다 적게 입력하면 저장이 거부되고 안내 메시지가 표시된다(decisions.md D-15).
- [ ] 예약일 생성/수정 시 시작 시간과 종료 시간을 입력할 수 있고, 종료 시간이 시작 시간보다 빠르거나 같으면 저장이 거부된다.
- [ ] 공개 목록/상세, 관리자 목록/상세 화면 모두에서 예약일의 시작~종료 시간이 표시된다.
- [x] 분리 슬롯 모드 예약일의 확정/대기 인원 표시는 연/캐주얼 풀별 세부 인원까지 함께 보여준다(관리자 상세: "확정 인원 N명(연 A/한도 · 캐 B/한도)" 형태, 공개 상세: "멤버 N명 확정(대기 M) · 캐주얼 X명 확정(대기 Y)" 형태). 통합 슬롯 모드에서는 풀 구분 없이 합계만 표시된다.

## 3. 연 멤버 관리

- [x] 이름+전화번호(정규화 후 기준)가 동일한 연 멤버를 중복 등록하면 오류가 발생한다.
- [x] 이름만 같고 전화번호가 다르면 별개 레코드로 등록 가능하다(오탐 방지 확인).
- [x] 연 멤버를 비활성화하면 이후 신규 예약 판별 시 캐주얼로 처리되고, 자동 배정 대상에서도 제외된다.
- [x] 연 멤버 "삭제" 액션은 레코드를 물리적으로 삭제하지 않고 `isActive=false`로만 처리한다(하드 삭제 없음).
- [x] 사용자 화면 어디에도 연 멤버 여부를 직접 선택하는 UI가 없다.
- [x] 관리자 화면에서 연 멤버 목록을 이름/활성여부 등으로 조회할 수 있다.

## 4. 월 멤버 관리

- [x] 같은 연 멤버를 같은 달의 서로 다른 요일(예: 화/목)로 각각 등록할 수 있다.
- [x] 동일한 연 멤버+연도+월+요일 조합을 중복 등록하면 거부된다.
- [x] 월 멤버 등록 화면에서 대상은 반드시 기존 연 멤버 중에서 선택하며, 신규 이름을 직접 입력해 등록할 수 없다.
- [x] 월 멤버를 비활성화하면 이후 자동 배정 대상에서 제외된다(연 멤버는 활성 상태여도 월 멤버 자체의 비활성화만으로 제외됨을 확인).
- [x] 월 멤버 목록의 각 행에는 "비활성화/활성화"와 "삭제"가 별도 버튼으로 존재한다. "비활성화"는 `isActive=false`로 토글만 하고 레코드를 남기며(재활성화 가능), "삭제"는 확인 대화상자 후 레코드를 실제로 완전히 제거한다(decisions.md D-26 — DB에서 직접 조회해도 행이 사라짐을 확인).
- [x] 관리자는 연도/월 단위로 월 멤버 목록을 필터링해 조회할 수 있다.
- [x] 월 멤버 목록에는 연 멤버 이름뿐 아니라 전화번호도 함께 표시된다.
- [x] 같은 연 멤버가 여러 요일에 등록되어 있으면 목록에서 이름/전화번호는 한 번만 표시되고, 등록된 요일별로 행이 나뉘어 어떤 요일에 배정되어 있는지 한눈에 구분된다.
- [x] 월 멤버 목록의 각 행에서 "수정" 버튼으로 적용 연도/월/요일/메모를 인라인으로 변경할 수 있다(decisions.md D-21). 대상 연 멤버 자체는 바꿀 수 없다.
- [x] 수정 후 조합(연 멤버+연도+월+요일)이 다른 레코드와 중복되면 등록 때와 동일하게 거부되고, 수정 모드가 유지된 채 에러 메시지가 표시된다.
- [x] 월 멤버 등록 폼에서 연 멤버를 한 번만 선택하고 적용 요일 체크박스를 여러 개(예: 월/수/금) 동시에 선택해 제출하면, 한 번의 등록으로 선택한 요일 수만큼 월 멤버 레코드가 한꺼번에 생성된다(decisions.md D-25).
- [x] 위 다중 요일 등록 시 일부 요일이 이미 등록되어 있으면 그 요일만 건너뛰고(중복 사유와 함께 안내) 나머지 요일은 정상 등록된다(부분 성공).

## 5. 월 멤버 자동 배정

- [x] 예약일 생성 폼을 제출하면 "월 멤버를 자동으로 추가하시겠습니까?" 확인 대화상자가 뜨고, 확인을 누르면 해당 연/월/요일과 일치하는 활성 월 멤버가 자동으로 예약에 추가된다(decisions.md D-19).
- [x] 확인 대화상자에서 취소를 누르면 이 예약일에는 월 멤버가 자동 배정되지 않고, "자동 배정을 건너뛰었습니다" 안내가 표시된다. 이후 예약일 상세 화면의 수동 "월 멤버 자동 배정" 버튼으로 나중에 실행할 수 있다.
- [x] 같은 요일에 세션이 두 개 이상 있을 때, 첫 세션 생성 시에는 자동 배정을 확인하고 두 번째 세션 생성 시에는 취소를 선택하면 월 멤버가 두 세션 모두에 중복 배정되지 않는다.
- [x] 예약일(예: 2026-07-07, 화)을 생성하면 해당 연/월/요일과 일치하는 활성 월 멤버가 자동으로 예약에 추가된다.
- [x] 월 멤버는 활성인데 연결된 연 멤버가 비활성인 경우 자동 배정되지 않는다.
- [x] 슬롯이 부족한 상태에서 자동 배정되는 월 멤버는 `WAITING`으로 생성된다.
- [x] 동일 예약일에 이미 확정/대기 중인 동일 이름+전화번호 예약이 있으면 자동 배정이 중복 생성하지 않는다. 이 판정은 source와 무관하다 — 예를 들어 연 멤버가 비활성 상태일 때 본인이 셀프 신청(source=USER, memberType=CASUAL)해 이미 확정된 상태라면, 이후 연 멤버가 다시 활성화되어도 자동 배정이 같은 사람을 중복 생성하지 않는다.
- [x] 동일 예약일에 취소 이력만 있는 동일인도 자동 배정 시 새 예약이 재생성된다(소스 무관, 스킵하지 않음).
- [x] 예약일 상세 화면의 "월 멤버 자동 배정" 버튼을 재실행해도 이미 `WAITING`/`CONFIRMED` 상태로 배정된 인원은 재생성(중복)되지 않는다(재실행 안전성). 단, `CANCELLED` 상태였던 인원은 재실행 시 새로 생성된다.
- [x] 특정 월 멤버를 앞으로 특정 요일 자동 배정에서 영구 제외하려면 취소가 아니라 해당 월 멤버 등록을 비활성화해야 하며, 비활성화 시 이후 자동 배정 대상에서 실제로 빠진다.
- [x] 자동 배정 실행 후 생성 건수/스킵 건수가 관리자 화면(예약일 생성 폼)에 표시된다.
- [x] 월 멤버 관리 화면에는 특정 연/월 전체 예약일에 소급 일괄 배정하는 버튼이 없다(decisions.md D-20 — 예약일 생성 시점 확인(D-19)과 예약일 상세의 수동 버튼 두 가지로 충분해 제거됨).
- [x] 월 멤버 관리 화면의 요일 선택은 목록을 클라이언트에서 즉시 필터링하는 용도로 동작한다 — 특정 요일을 선택하면 그 요일에 등록된 월 멤버만 표시되고, "전체 요일"을 선택하면 다시 전체 목록이 표시된다(decisions.md D-20).
- [x] 새 월 멤버 등록 폼을 제출하면 "기존 예약일에도 자동으로 추가하시겠습니까?" 확인 대화상자가 뜨고, 확인을 누르면 등록한 연/월/요일과 일치하는 기존 예약일 전체에 소급 배정되며 생성/스킵 건수가 표시된다(decisions.md D-22).
- [x] 위 확인 대화상자에서 취소를 누르면 새 월 멤버는 등록되지만 기존 예약일에는 반영되지 않는다(신규 예약일에는 D-19 트리거로 정상 반영됨).
- [x] 기존 예약일 소급 배정 대상에는 이미 종료된 예약일(decisions.md D-23)이 포함되지 않는다 — 등록한 연/월/요일과 일치해도 종료된 예약일에는 배정되지 않는다.

## 6. 사용자 예약 신청

- [ ] 이름/전화번호 외의 추가 정보 입력 없이 예약 신청이 완료되며, `memberType`을 직접 선택하는 입력 요소는 존재하지 않는다.
- [ ] 이름과 전화번호가 모두 일치하는 활성 연 멤버로 신청하면 `memberType=ANNUAL`로, 하나라도 다르면 `CASUAL`로 자동 판별된다(분리 모드에서는 각각의 슬롯 풀 기준으로 판단).
- [ ] 동일 예약일에 이미 확정/대기 중인 동일인이 재신청하면 "이미 예약됨" 안내와 함께 거부된다.
- [ ] 동일 예약일에 취소 이력만 있는 동일인은 재신청(새 예약 생성)이 가능하다.
- [ ] 슬롯이 가득 찬 상태에서 신청하면 `WAITING`으로 등록되고 대기 상태 안내를 받는다.
- [ ] 신청 직후 화면에서 본인 예약이 확정인지 대기인지 명확히 구분되어 표시된다.
- [ ] 공개 화면에서 같은 예약일의 다른 예약자 이름과 상태(확정/대기)는 보이지만 전화번호는 어디에도 노출되지 않는다.
- [ ] 예약일의 종료 시각(`date+endTime`, Pacific/Auckland 기준)이 지나면 목록/상세에 "종료됨" 배지가 표시되고, 상세 화면의 예약 신청 폼은 안내 문구로 대체되어 더 이상 신청할 수 없다(decisions.md D-23). API를 직접 호출해도(`POST /api/bookings`) `CONFLICT` 에러로 거부된다.
- [ ] 종료된 예약일도 목록/상세에서 계속 조회 가능하다(숨겨지지 않음) — 예약자 명단도 그대로 보인다.
- [x] 예약 가능일 목록 화면에 시작일/종료일 날짜 필터가 있고, 범위를 설정하면 해당 기간의 예약일만 표시된다. 기본값은 오늘부터 7일 후까지이며, 값을 바꾸면 "초기화" 버튼이 나타나 기본값으로 되돌릴 수 있다(decisions.md D-24).

## 7. 예약 취소 (사용자)

- [ ] 전화번호만 입력하면 해당 번호로 등록된 모든 예약(여러 날짜/여러 이름 포함)이 목록으로 조회된다.
- [ ] 목록에서 bookingId 단위로 개별 취소를 선택할 수 있다.
- [ ] 취소 요청 시 입력한 전화번호가 해당 bookingId의 전화번호와 다르면 취소가 거부된다.
- [ ] 취소된 예약은 DB에서 삭제되지 않고 상태만 `CANCELLED`로 변경되며 취소 시각(`cancelledAt`)이 기록된다.
- [ ] `WAITING` 예약을 취소해도 다른 대기자의 순번/상태에는 영향이 없다.
- [ ] 취소 후 같은 사용자가 같은 예약일에 다시 신청하면 새로운 예약 건으로 정상 생성된다.
- [ ] 예약일이 이미 종료됐으면(decisions.md D-23) 내 예약 조회/취소 화면에서 해당 예약 행의 취소 버튼이 "종료된 예약일 (취소 불가)" 텍스트로 대체되어 취소할 수 없다. API를 직접 호출해도(`POST /api/bookings/[id]/cancel`) `CONFLICT` 에러로 거부된다.
- [x] 내 예약 조회/취소 화면의 조회 결과에도 시작일/종료일 날짜 필터가 있고, 기본값은 "오늘부터 무제한"이다(상한 없음) — 오늘보다 훨씬 뒤 날짜의 예약도 기본 상태에서 조회 결과에 포함된다. 지난 예약만 기본으로 가려진다. 전화번호로 새로 조회하면 필터가 이 기본값으로 재설정된다(decisions.md D-24).
- [ ] 관리자의 취소(`adminCancelBooking`, 예약일 관리 화면의 취소 액션)는 예약일 종료 여부와 무관하게 항상 가능하다.

## 8. 대기열 관리 및 자동 승격

- [ ] 슬롯이 가득 찬 상태의 신규 신청은 항상 `WAITING`으로 생성된다.
- [ ] 슬롯 20명 확정, 대기 3명 상태에서 슬롯을 22명으로 늘리면 대기 2명이 신청 순서대로 자동 확정된다.
- [ ] 슬롯 증가로 승격될 인원이 여러 명일 때, 먼저 신청한 사람이 우선 승격되며, 신청 시각이 같으면 id가 작은 예약이 우선한다.
- [ ] `CONFIRMED` 예약 취소(사용자/관리자 무관) 시 대기 1순위 1명만 자동 확정되고 나머지 대기자는 대기 상태를 유지한다.
- [ ] 분리 슬롯 모드에서 `ANNUAL` 예약을 취소하면 `ANNUAL` 대기자만 승격되고 `CASUAL` 대기자는 영향받지 않으며, 반대로 `CASUAL` 예약을 취소해도 `ANNUAL` 대기자는 영향받지 않는다(취소로 인한 승격도 풀 간 교차 없음, 슬롯 증가 시나리오와 동일 규칙).
- [ ] 취소 시점에 같은 풀에 대기자가 없으면 승격은 일어나지 않지만 슬롯은 정상적으로 비워져, 이후 새로운 신청자가 대기 없이 바로 `CONFIRMED`로 등록된다(슬롯 자동 복구 확인).
- [ ] 관리자가 대기 예약을 개별 승인할 때, 슬롯 여유가 없어도 항상 승인되며 부족한 만큼 슬롯 수(분리 모드면 해당 memberType의 세부 슬롯 + totalSlots)가 자동으로 늘어난다(decisions.md D-14). 사용자가 직접 신청하는 경우에는 이 자동 확장이 적용되지 않는다.
- [ ] 관리자의 개별 승인은 대기 순번을 건너뛰어도(FIFO 미준수) 허용된다.
- [ ] 분리 슬롯 모드에서 연 슬롯에 여유가 생겨도 캐주얼 대기자는 승격되지 않고, 반대의 경우도 동일하다(풀 간 교차 승격 없음).
- [ ] 통합 슬롯 모드에서는 연/캐주얼 구분 없이 전체 대기열 순서로 승격된다.

## 9. 관리자 예약 운영

- [ ] 관리자 화면에서 예약일별 예약자 목록에 이름/전화번호/상태/유형(ANNUAL·CASUAL)/source가 모두 표시된다.
- [ ] 관리자가 대기 예약을 확정으로 변경하면 항상 성공하며, 슬롯이 부족했던 경우 슬롯 수가 자동으로 늘어나 있다.
- [ ] 관리자가 예약을 취소 처리하면 `cancelledAt`이 기록되고 대기자 자동 승격 로직이 사용자 취소와 동일하게 실행된다.
- [ ] 관리자가 직접 추가한 예약은 `source=ADMIN`으로 기록되며, 이름+전화번호 기준 중복/재예약 규칙과 memberType 판정을 동일하게 적용받는다. 슬롯이 부족하면 자동으로 슬롯을 늘려 확정 처리한다(WAITING으로 생성되지 않음).

- [x] 다른 예약일의 QR을 스캔하면 "이 QR 코드는 다른 예약일의 예약입니다" 오류로 거부된다.

- [x] 종료된 예약일(D-23)에서도 관리자의 입장/퇴장/초기화 처리는 차단되지 않는다.

## 11. 비기능 테스트

### 동시성 테스트
- [ ] 동일 예약일에 잔여 슬롯 1자리를 두고 동시에 2건 이상의 예약 요청을 보냈을 때, 슬롯 초과 없이 정확히 1건만 `CONFIRMED`가 되고 나머지는 `WAITING`이 된다.
- [ ] 동일 대기자 목록에 대해 관리자의 슬롯 증가 요청과 사용자의 취소 요청이 거의 동시에 발생해도, 승격 대상이 중복 확정되거나 슬롯이 초과되지 않는다.
- [ ] 동일 이름+전화번호로 동일 예약일에 대한 예약 요청을 동시에 2건 보냈을 때, 중복 예약(둘 다 WAITING/CONFIRMED로 생성)이 발생하지 않는다.

### 트랜잭션 롤백 테스트
- [ ] 예약 생성 트랜잭션 중간(예: 슬롯 카운트 조회 이후 insert 이전)에 강제 오류를 주입했을 때, 부분적으로 반영된 데이터 없이 전체가 롤백된다.
- [ ] 대기자 자동 승격 처리 도중 일부 레코드 업데이트에서 실패가 발생하면, 이미 처리된 승격 건도 함께 롤백되어 상태가 일관되게 유지된다.
- [ ] 월 멤버 자동 배정(`applyMonthlyMembersToMonth`) 중 특정 예약일 처리에서 실패가 발생해도, 해당 예약일 단위 트랜잭션만 롤백되고 다른 예약일의 처리 결과는 영향받지 않는다.

### 타임존 경계 테스트 (Pacific/Auckland)
- [ ] 뉴질랜드 자정(00:00 NZST/NZDT) 직전/직후에 생성한 예약일의 `dayOfWeek`가 UTC 기준이 아닌 Pacific/Auckland 기준으로 정확히 계산된다.
- [ ] 서머타임 전환(NZDT↔NZST) 전후 날짜에 대해서도 날짜/요일 계산이 어긋나지 않는다.
- [ ] 대시보드의 "오늘" 판정이 자정 경계(Pacific/Auckland 기준)에서 하루 앞/뒤로 밀리지 않는다.

### PII 암호화 테스트 (전화번호)
- [ ] DB(Turso)를 직접 조회했을 때 `phone`/`normalizedPhone` 평문 컬럼이 존재하지 않고 `phoneHash`/`phoneEncrypted`만 저장되어 있다.
- [ ] 같은 전화번호로 다시 예약/등록해도 `phoneHash` 값이 항상 동일하다(결정론적 확인).
- [ ] `phoneEncrypted` 값은 같은 전화번호라도 저장할 때마다 다른 값이 나온다(랜덤 IV 확인), 그러나 복호화 시 항상 같은 평문으로 복원된다.
- [ ] 관리자 예약자 목록/연 멤버 목록 API 응답에는 복호화된 평문 전화번호만 담기고, `phoneHash`/`phoneEncrypted` 원값은 응답에 노출되지 않는다.
- [ ] 사용자 취소(`cancelBooking`), 전화번호 조회(`lookupBookingsByPhone`), 멤버 판정(`determineMemberType`) 모두 복호화 없이 `phoneHash` 비교만으로 정상 동작한다.
- [ ] `PII_SECRET_KEY`가 설정되지 않은 상태로 앱을 기동하면 명확한 에러로 즉시 실패한다(암호화 키 없이 조용히 평문 저장되는 경우가 없어야 함).

### 정규화 엣지케이스
- [ ] 전화번호에 하이픈, 공백, 괄호, 국가번호(+64 등)가 섞여 입력되어도 숫자만 남는 정규화가 일관되게 동작한다.
- [ ] 이름 앞뒤 공백만 다른 두 입력값이 동일 인물로 정규화되어 처리된다(trim 확인).
- [ ] 빈 문자열 또는 공백만 있는 이름/전화번호 입력에 대한 처리(허용 여부, 오류 메시지)가 일관된다.
- [ ] 지나치게 짧은 전화번호(예: 3자리)나 숫자가 전혀 없는 입력에 대한 동작이 예상대로(허용 또는 명확한 오류) 처리된다.

## 13. 배포 전 최종 점검

- [ ] 환경변수 `ADMIN_PASSWORD`가 Vercel 프로덕션 환경변수에 설정되어 있고, 기본값/예시값이 그대로 남아있지 않다.
- [ ] 세션 서명용 비밀키(`ADMIN_SESSION_SECRET`)가 설정되어 있고 `ADMIN_PASSWORD`와 별도 값으로 관리된다.
- [ ] `PII_SECRET_KEY`가 Vercel 프로덕션 환경변수에 설정되어 있다(전화번호 암/복호화용, 누락 시 전화번호 저장 자체가 실패해야 함).
- [ ] `TURSO_DATABASE_URL`, `TURSO_AUTH_TOKEN`이 Vercel 프로덕션 환경변수에 설정되어 있고, Turso 대시보드에서 해당 DB가 정상 상태인지 확인한다.
- [ ] 배포 전 스키마 변경사항이 `prisma migrate diff` + Turso CLI로 실제 프로덕션 DB에 반영되어 있다(로컬 마이그레이션과 원격 DB 스키마가 어긋나지 않는지).
- [ ] Vercel 배포(프로덕션 브랜치 push 또는 수동 배포)가 정상적으로 완료되고, 빌드 로그에 에러가 없다.
- [ ] Vercel Hobby 플랜의 비상업적 이용 조건에 부합하는지 재확인한다(회비 결제·광고 등 상업적 요소 추가 시 플랜 재검토 필요).
- [ ] 서버 타임존 설정과 무관하게 애플리케이션이 Pacific/Auckland 기준으로 날짜를 계산하는지 재확인한다(서버리스 함수의 런타임 타임존에 의존하지 않는지).
- [ ] 프로덕션에서 쿠키에 `secure` 속성이 적용되어 HTTPS 환경에서만 전송되는지 확인한다.
- [ ] 관리자 페이지 및 관리자 API 전 구간이 미들웨어로 보호되어 미인증 접근이 차단되는지 최종 확인한다.
- [ ] 에러 응답에 스택 트레이스 등 민감한 내부 정보가 노출되지 않는지 확인한다.
