# 배드민턴 예약 관리 시스템 — 결정 이력 (ADR)

기준: 개발자와 사용자가 기획/설계 단계에서 논의해 확정한 주요 의사결정 모음.
`requirements.md`에는 최종 결론만 기재되어 있고 검토 과정의 대안은 남아있지 않으므로, 이 문서가 "왜 이렇게 정했는지"에 대한 배경 기록을 담당한다.
날짜는 별도로 기재하지 않으며, 결정 번호(D-01, D-02, ...)로 식별한다.

## D-01. 취소 후 재예약 정책의 적용 범위

**배경**
원본 요구사항에는 "취소 이력이 있으면 자동 재예약하지 않고 관리자가 처리한다"는 조항이 있었다. 그러나 이 조항이 사용자가 셀프 서비스로 다시 예약을 신청하는 일반 예약 API에도 적용되는지, 아니면 월 멤버 자동 배정처럼 시스템이 자동으로 예약을 생성하는 경로에만 적용되는지가 불명확했다.

**검토한 대안**
- (A) 사용자 셀프 재예약 허용, `MONTHLY_MEMBER_AUTO` 소스에만 차단 규칙 적용
- (B) `source`와 무관하게 취소 이력이 있으면 완전 차단(모든 경우 관리자 개입 필요)

**최종 결정**
(A) 채택. `USER`, `ADMIN` 소스는 `CANCELLED` 이력이 있어도 새 예약 행 생성을 허용하고, `MONTHLY_MEMBER_AUTO` 소스만 차단한다.

**결정 사유**
일반적인 예약 시스템에서 사용자가 스스로 취소한 자리를 본인이 다시 신청하지 못하게 막는 것은 통상적인 UX 기대에 어긋난다. 반면 월 멤버 자동 배정은 시스템이 능동적으로 만드는 예약이므로, 과거에 취소된 이력이 있다면(예: 해당 월 멤버가 그날 불참 의사를 표시한 것일 수 있음) 무조건 재생성하지 않고 관리자가 상황을 보고 판단하도록 남겨두는 것이 안전하다.

> **개정 안내**: 위 문단의 `MONTHLY_MEMBER_AUTO` 예외 조항은 이후 **D-08**에서 재검토되어 폐지되었다. 현재는 소스와 관계없이 `CANCELLED` 이력이 재예약을 막지 않는다. D-08 참고.

## D-02. 같은 날짜의 다중 예약 세션 허용 여부

**배경**
예약일(BookingDay)에 `date` 필드에 유니크 제약을 둘지, 즉 하루에 하나의 예약 세션만 허용할지 여러 세션(예: 오전/오후, 서로 다른 장소)을 허용할지가 불명확했다.

**검토한 대안**
- (A) 날짜당 1개 세션만 허용, `date`에 unique 제약
- (B) 날짜당 여러 세션 허용, `date`에 unique 제약 없음(대신 `label` 필드로 구분)

**최종 결정**
(B) 채택. `BookingDay.date`에는 unique 제약을 두지 않고, `label`(optional)로 세션을 구분한다.

**결정 사유**
동호회 운영 특성상 같은 날 여러 시간대나 장소에서 별도로 예약을 받아야 하는 상황(예: 오전 세션/오후 세션)이 실제로 발생할 수 있다. 처음부터 다중 세션을 허용해 두는 것이 이후 운영 유연성 측면에서 더 안전하며, unique 제약을 두면 나중에 이를 완화하기 위한 마이그레이션 비용이 더 크다.

## D-03. 예약 취소 방식: 전화번호 단독 취소 vs 2단계 조회-선택 취소

**배경**
전화번호만으로 즉시 취소를 처리하면, 같은 전화번호로 여러 건(다른 이름, 다른 날짜)의 예약이 존재할 때 어떤 예약을 취소해야 하는지 특정할 수 없는 문제가 있었다.

**검토한 대안**
- (A) 전화번호로 본인 예약 목록을 먼저 조회한 뒤, 사용자가 그중 하나를 선택해 취소
- (B) 이름+전화번호+날짜 조합으로 즉시 취소 처리(조회 단계 없이 한 번에 취소)

**최종 결정**
최종적으로 채택한 방식은 (A)를 구체화한 형태로, "`bookingId` + 전화번호를 받아 전화번호 일치 여부를 검증한 뒤 취소 처리"하는 API를 만들고, 이를 위한 사전 단계로 전화번호 기반 예약 목록 조회(lookup) 엔드포인트를 별도로 둔다. 즉 조회(`lookupBookingsByPhone`) → 사용자가 `bookingId` 선택 → 취소(`cancelBooking(bookingId, phone)`) 2단계 플로우로 확정되었다.

**결정 사유**
이름+전화번호+날짜 조합만으로는 같은 사람이 같은 날짜에 이름을 다르게 입력했거나(오타 등) 여러 세션에 각각 예약한 경우를 여전히 특정하지 못할 수 있다. `bookingId` 단위로 명확하게 취소 대상을 지정하면서도, 취소 시점에 전화번호 일치 여부를 재검증해 다른 사람의 예약을 임의로 취소하지 못하도록 하는 것이 더 안전하고 명확하다.

## D-04. 공개 페이지에서의 예약자 이름 노출 범위

**배경**
같은 예약일에 다른 예약자들의 이름을 공개 화면에서 어디까지 보여줄지 불명확했다. 개인정보 노출 우려와, 예약 현황을 투명하게 보여줘야 하는 운영상 필요성 사이의 균형이 필요했다.

**검토한 대안**
- (A) 이름 일부 마스킹(예: "김*수")
- (B) 인원수만 표시하고 이름은 비공개
- (C) 이름 전체 공개(단, 전화번호는 비공개 유지)

**최종 결정**
(C) 채택. 같은 예약일의 다른 예약자 이름은 전체 공개하며, 전화번호는 항상 비공개로 유지한다.

**결정 사유**
이 시스템은 동호회 전용 예약 시스템으로, 예약자들이 서로 아는 사이인 경우가 대부분이라 이름을 숨길 실익이 적고, 오히려 "누가 오는지"를 아는 것이 참석 여부 조율 등에 실질적으로 도움이 된다. 반면 전화번호는 개인정보 성격이 강하고 노출될 경우 원치 않는 연락으로 이어질 수 있어 비공개를 유지한다.

## D-05. 서비스 기준 타임존

**배경**
날짜/요일 계산 및 "오늘" 대시보드 기준이 되는 타임존이 처음에는 명확히 지정되지 않았다.

**검토한 대안**
- (초기 제안) Asia/Seoul을 기본값으로 사용
- (변경 후 확인된 사실) 실제 서비스 지역은 뉴질랜드

**최종 결정**
Pacific/Auckland(IANA 타임존 이름 기준)로 확정. 뉴질랜드는 서머타임(NZDT/NZST)을 사용하므로 고정 UTC 오프셋이 아닌 IANA 타임존 이름 기반으로 날짜/요일을 계산한다.

**결정 사유**
최초 논의 시점에는 서비스 지역에 대한 명확한 정보 없이 통상적인 기본값(Asia/Seoul)을 임시로 제안했으나, 이후 사용자가 실제 운영 지역이 뉴질랜드임을 확인해주었다. 뉴질랜드는 서머타임을 사용하는 지역이므로, 고정 offset(예: UTC+12)으로 처리하면 서머타임 전환 시기에 날짜/요일 계산이 어긋난다. 따라서 IANA 타임존 이름(Pacific/Auckland)을 사용해 서머타임 전환을 자동으로 반영하도록 결정했다.

## D-06. 별도 이견 없이 확정된 기본값 모음

**배경**
아래 항목들은 논의 과정에서 뚜렷한 대안 검토 없이, 요구사항 정의 단계에서 합리적인 기본값으로 바로 합의된 사항들이다. 각각은 사소해 보이지만 구현 시 임의로 다르게 해석될 여지가 있어 명시적으로 기록해 둔다.

**최종 결정 목록**
1. `BookingDay.dayOfWeek`는 사용자 입력 항목이 아니며, 서버가 `date`로부터 자동 계산한다.
2. 이름은 `trim`(앞뒤 공백 제거), 전화번호는 숫자만 남기는 정규화 규칙을 일괄 적용한다.
3. `SEPARATED` 슬롯 모드는 저장 시 `totalSlots = annualSlots + casualSlots` 검증을 수행하며, `COMBINED` 모드는 세부 슬롯(`annualSlots`/`casualSlots`)을 사용하지 않는다(0으로 둠).
4. 분리 슬롯 모드에서 연 ↔ 캐주얼 간 교차 승격은 없다. 각 풀(연 슬롯/캐주얼 슬롯) 내에서만 FIFO 승격이 이루어진다.
5. 월 멤버 자동 배정은 항상 `memberType=ANNUAL`로 처리하며, `MonthlyMember.isActive`와 연결된 `AnnualMember.isActive`를 모두 확인해 하나라도 비활성이면 배정에서 제외한다.
6. 관리자 세션은 별도 DB 세션 테이블 없이 서명된 HttpOnly 쿠키로 관리하며, 만료 예시는 24시간이다.
7. 관리자의 개별 대기 예약 승인은 슬롯 여유만 확인하며 FIFO 순서를 강제하지 않는다(FIFO는 자동 승격 로직에만 적용된다).
8. `AnnualMember`는 `normalizedName + normalizedPhone` 조합에 유니크 제약을 둬 중복 등록을 방지한다.

**결정 사유**
이 항목들은 상충하는 대안이 제시되지 않고 요구사항 정의 초기부터 합리적 기본값으로 채택되었으나, 구현자가 임의로 다른 방식(예: 클라이언트에서 dayOfWeek를 입력받는다거나, 정규화 규칙을 다르게 적용하는 등)으로 해석하지 않도록 결정 사항으로 명시해 둔다.

## D-07. 연/월 멤버 "삭제"의 의미 — 비활성화로 통일

**배경**
원본 요구사항(4·5·15번)은 "등록/수정/삭제"라는 표현을 사용했지만, 데이터 모델(17번)에는 `isActive` 필드만 있고 하드 삭제 지원 여부에 대한 언급이 없었다. 기획서.md 검토 과정에서도 이 지점이 미해결 이슈로 지적되었다(이슈 1).

**검토한 대안**
- (A) 관리자 화면의 "삭제" 액션이 실제로 레코드를 물리적으로 삭제(하드 삭제)
- (B) "삭제" 액션은 실제로는 비활성화(`isActive=false`)만 수행하고, 하드 삭제는 지원하지 않음

**최종 결정**
(B) 채택. `AnnualMember`, `MonthlyMember` 모두 하드 삭제를 지원하지 않는다. 관리자 화면/API의 "삭제" 액션은 내부적으로 `isActive=false` 처리로 구현한다.

**결정 사유**
`AnnualMember`는 `Booking.matchedAnnualMemberId`와 `MonthlyMember.annualMemberId`에서 참조되므로, 하드 삭제를 허용하면 참조 무결성이 깨지거나(외래키 오류) 과거 예약 이력에서 "그때는 연 멤버였는지" 여부를 더 이상 알 수 없게 된다. 비활성화만 지원하면 이력을 보존하면서도 향후 자동 배정/멤버 판별 대상에서 제외한다는 실질적 목적을 그대로 달성할 수 있다. UI 문구는 "삭제"를 유지하되, 실제 동작은 비활성화임을 관리자에게 안내하는 것을 권장한다(예: "삭제하면 목록에서 숨겨지며 언제든 다시 활성화할 수 있습니다").

## D-08. 월 멤버 자동 배정의 취소 이력 처리 — D-01 개정

**배경**
D-01에서는 `CANCELLED` 이력이 있을 때 `USER`/`ADMIN` 소스는 재예약을 허용하되, `MONTHLY_MEMBER_AUTO` 소스만 예외적으로 재생성을 막고 관리자가 직접 처리하도록 정했다. 이후 이 예외가 실제로 필요한 안전장치인지 재검토가 이루어졌다.

**검토한 대안**
- (A) D-01 그대로 유지 — `MONTHLY_MEMBER_AUTO`만 `CANCELLED` 시 스킵, 관리자 수동 처리
- (B) 소스와 관계없이 `CANCELLED`는 재예약을 막지 않도록 통일 — 자동 배정도 다음 실행 시 항상 재생성

**최종 결정**
(B) 채택. `createBooking`과 `applyMonthlyMembersToBookingDay`는 이제 source와 무관하게 같은 `bookingDayId + normalizedName + normalizedPhone` 조합에 `CANCELLED`만 있으면 새 예약을 생성한다. D-01의 `MONTHLY_MEMBER_AUTO` 예외 조항은 이 결정으로 대체된다.

**결정 사유**
특정 월 멤버를 앞으로 계속 자동 배정에서 제외하고 싶다는 요청은 "그 날짜 하나의 취소"가 아니라 "월 멤버 등록 자체"에 대한 변경 요청이다. 따라서 취소 이력으로 자동 배정을 스킵하는 방식보다, 월 멤버 등록을 비활성화(D-07)하는 방식이 의도를 더 명확하고 정확하게 반영한다. 반대로 취소 이력 기반 스킵을 유지하면, 관리자가 재실행할 때마다 "이 취소가 영구 제외 의사인지 단순 변심인지"를 매번 판단해야 하는 불필요한 부담이 생긴다. 규칙을 D-01과 동일하게 통일함으로써 `createBooking`의 중복 판정 로직이 소스에 관계없이 단일해지고, 구현과 테스트가 단순해진다(MVP 단순성 원칙에 부합).

## D-09. 배포 방식 — Docker/자체 서버 대신 Vercel + Turso

**배경**
원본 요구사항(21번)은 "Docker 지원, SQLite 파일 볼륨 마운트 지원"을 배포 방식으로 명시했다. 이후 사용자가 "최대한 무료로 운영하고 싶다"는 제약을 추가로 제시하면서, 이 배포 방식이 실제로 무료로 지속 가능한지 재검토가 필요해졌다. GitHub Pages는 정적 호스팅이라 서버 코드/DB를 아예 실행할 수 없어 처음부터 제외되었다.

**검토한 대안**
- (A) 기존 계획 유지 — Oracle Cloud "Always Free" VM에 Docker + SQLite 파일(볼륨 마운트)로 자체 호스팅. 비용은 $0이지만 VM 직접 관리(OS 패치, 보안)가 필요하고, 가입 심사 지연/거절, 무료 한도 축소(2026년 6월부터 축소) 등 리스크가 있음.
- (B) Vercel(Hobby, 무료) + Turso(libSQL, 무료 티어) 조합. 서버/컨테이너를 직접 관리하지 않아도 되며, Prisma가 libSQL 드라이버 어댑터를 공식 지원해 스키마 변경 없이 연결 방식만 바꾸면 됨. 대신 로컬 SQLite 파일 대신 외부 관리형 DB(Turso)에 데이터가 저장되고, Docker는 사용하지 않게 됨.

**최종 결정**
(B) 채택. Vercel(Hobby 플랜)에 Next.js 앱을 배포하고, DB는 Turso(무료 티어)를 사용한다. `Dockerfile`/`docker-compose.yml` 기반 배포는 더 이상 사용하지 않는다.

**결정 사유**
동호회 예약 시스템 규모(소규모 트래픽, 소량 데이터)에서는 Turso 무료 티어(5GB 저장, 월 5억 행 읽기, 월 1천만 행 쓰기)로 사실상 무제한에 가깝게 여유롭고, Vercel Hobby 플랜도 비상업적 개인/동호회 프로젝트 용도에 부합한다. 반면 자체 서버(A안)는 "무료"이긴 하지만 서버 보안 패치, 장애 대응, 가입 승인 불확실성 등 운영 부담이 실질적으로 존재해 MVP 단계에서 감당할 이유가 적다고 판단했다. 다만 이 결정으로 인해 요구사항 21번의 "Docker 지원" 조항은 폐기되고, 배포 구성(마이그레이션 워크플로우 포함)은 별도 `deployment.md`에 정리했다(D-12 참고).

## D-10. 전화번호 저장 방식 — 평문 대신 phoneHash + phoneEncrypted

**배경**
D-09에서 DB를 로컬 SQLite 파일이 아닌 외부 관리형 서비스(Turso)에 두기로 하면서, 전화번호(PII)를 평문으로 저장하는 것이 적절한지 재검토가 필요해졌다. 동시에 `determineMemberType`, 중복 예약 확인, 취소 시 본인확인, 전화번호로 예약 조회 등 여러 로직이 전화번호의 "일치 여부" 비교에 의존하고 있어, 단순히 필드를 암호화하면 이 조회들이 깨질 수 있다는 문제가 있었다.

**검토한 대안**
- (A) 평문 저장 유지 (기존 `phone`/`normalizedPhone` 컬럼 그대로)
- (B) 필드 전체를 AES 등으로 암호화만 하고 조회 시 애플리케이션에서 전체 로우를 복호화해 비교 (조회 성능/확장성 나쁨, 데이터가 늘어나면 사실상 불가능)
- (C) 조회/비교용 `phoneHash`(HMAC-SHA256, 결정론적)와 열람용 `phoneEncrypted`(AES-256-GCM, 복호화 가능)를 분리 저장. 모든 equality 조회는 `phoneHash`로, 관리자가 실제 번호를 봐야 할 때만 `phoneEncrypted`를 복호화

**최종 결정**
(C) 채택. `AnnualMember`, `Booking`의 `phone`/`normalizedPhone` 컬럼을 `phoneHash` + `phoneEncrypted`로 대체한다. `determineMemberType`, `createBooking`의 중복확인, `cancelBooking`의 본인확인, `lookupBookingsByPhone`은 모두 `phoneHash` 비교로 동작하도록 변경한다(복호화 불필요). 이름은 공개 페이지에 그대로 노출되는 정보(D-04)이므로 암호화 대상에서 제외한다.

**결정 사유**
전화번호는 개인정보이며, 외부 관리형 DB에 평문으로 쌓아두는 것은 데이터 유출 시 피해가 크다. 그렇다고 단순 암호화(B안)를 하면 애플리케이션의 핵심 로직(중복확인, 멤버 판정, 취소 본인확인)이 전부 "저장된 값과 입력값이 같은가"를 물어야 하는데, 매번 랜덤 IV를 쓰는 암호화는 같은 입력이라도 다른 ciphertext를 만들어내 이 비교 자체가 불가능해진다. 결정론적 해시(HMAC)를 별도 컬럼으로 두면 이 문제가 해결되면서도, 해시만으로는 원래 번호를 복원할 수 없어 안전하다. 관리자가 실제로 회원에게 연락해야 하는 경우(비상 연락 등)를 위해 복호화 가능한 `phoneEncrypted`를 별도로 둔다. 두 값 모두 같은 마스터 키(`PII_SECRET_KEY`)에서 파생하되 용도별 서브키를 분리해 한쪽이 손상되어도 다른 쪽에 영향이 없도록 한다.

## D-11. 관리자 비밀번호 저장 방식 — 평문 환경변수 대신 salted hash

**배경**
원본 요구사항은 `ADMIN_PASSWORD` 환경변수에 비밀번호를 그대로 저장하고 문자열 비교로 로그인을 검증하는 방식이었다. DB 저장 방식을 재검토하는 김에, 환경변수에 평문 비밀번호를 두는 것도 같은 시각에서 재검토했다(Vercel 등 배포 플랫폼의 환경변수 설정 화면에 접근 권한이 있는 사람은 평문 비밀번호를 그대로 볼 수 있음).

**검토한 대안**
- (A) 기존 방식 유지 — `ADMIN_PASSWORD` 평문 저장, `crypto.timingSafeEqual`로 문자열 비교(타이밍 공격만 방지)
- (B) 비밀번호의 salted hash(`scrypt`)를 `ADMIN_PASSWORD_HASH` + `ADMIN_PASSWORD_SALT`로 저장하고, 로그인 시 입력값을 동일 방식으로 해시해 비교

**최종 결정**
(B) 채택. 관리자 비밀번호 원문은 코드베이스/환경변수 어디에도 저장하지 않는다. 최초 설정 시 1회성 스크립트로 해시를 생성해 환경변수에 등록한다.

**결정 사유**
관리자가 한 명뿐이고 세션도 단순한 서명 쿠키 방식이라 전체 인증 체계는 가볍게 유지하되, "평문 비밀번호가 어딘가에 그대로 적혀 있다"는 사실 자체는 피할 수 있는 리스크다. 배포 플랫폼(Vercel) 환경변수 화면, 로컬 `.env` 파일, 백업 등 여러 경로로 평문이 노출될 가능성을 생각하면, 해시로만 저장해도 로그인 기능에 미치는 영향 없이 리스크를 제거할 수 있다. `bcrypt` 같은 외부 라이브러리 대신 Node 내장 `crypto.scrypt`를 사용해 의존성을 늘리지 않는다(MVP 단순성 원칙에 부합, 코드 몇 줄로 충분).

> **개정 안내**: 이 결정은 **D-13**에서 재검토되어 폐기되었다. 비밀번호를 바꿀 때마다 해시 생성 스크립트를 실행해야 하는 설정 번거로움이, 관리자 1인 체제에서 얻는 보안 이득에 비해 크다고 판단해 평문 환경변수로 복귀했다. D-13 참고.

## D-12. 배포 관련 내용의 문서 분리 — architecture.md에서 deployment.md로 이동

**배경**
호스팅/배포 대상(Vercel, Turso 등)은 프로젝트 상황에 따라 이후 바뀔 가능성이 있는 반면, `architecture.md`는 도메인 로직을 어떤 코드 구조로 구현할지를 다루는 상대적으로 안정적인 문서다. 두 성격의 정보가 한 파일에 섞여 있으면, 배포 대상이 바뀔 때마다 코드 아키텍처와 무관한 부분 때문에 `architecture.md`를 계속 수정해야 하는 문제가 있다.

**검토한 대안**
- (A) `architecture.md`에 배포 내용을 계속 포함
- (B) 배포 관련 내용을 `deployment.md`로 분리하고, `architecture.md`에는 짧은 참조만 남김

**최종 결정**
(B) 채택. 호스팅 구성, 환경변수 목록, 마이그레이션 운영, 플랫폼별 주의사항은 `deployment.md`에서 관리한다. `architecture.md`는 디렉토리 구조/서비스 계층/스키마/API/인증/트랜잭션/에러 컨벤션 등 코드 구조만 다룬다.

**결정 사유**
배포 대상 변경(예: 이후 자체 서버로 이전, 다른 DB로 교체 등)이 발생해도 코드 아키텍처 문서를 건드릴 필요가 없어지고, 반대로 코드 구조가 바뀌어도 배포 문서를 건드릴 필요가 없어 두 문서의 변경 이력이 서로 섞이지 않는다.

## D-13. 관리자 비밀번호 저장 방식 재검토 — 평문 환경변수로 복귀 (D-11 폐기)

**배경**
D-11에서 관리자 비밀번호를 `ADMIN_PASSWORD_HASH`/`ADMIN_PASSWORD_SALT`(scrypt 해시)로 저장하기로 했으나, 실제 운영 관점에서 재검토했다. 비밀번호를 설정하거나 바꿀 때마다 별도 스크립트를 로컬에서 실행해 해시값을 생성한 뒤 Vercel 환경변수에 붙여넣어야 하는 절차가, 관리자 1인 체제의 소규모 프로젝트에 걸맞지 않게 번거롭다는 지적이 있었다.

**검토한 대안**
- (A) D-11 그대로 유지 — `ADMIN_PASSWORD_HASH`/`ADMIN_PASSWORD_SALT`, 비밀번호 변경 시마다 해시 생성 스크립트 실행 필요
- (B) `ADMIN_PASSWORD_HASH` 하나로 축소(`salt:hash` 결합 포맷) — env var 개수는 줄지만 스크립트 실행 자체는 여전히 필요
- (C) 평문 `ADMIN_PASSWORD` 환경변수로 복귀(원본 요구사항 방식) — 설정 번거로움 없음

**최종 결정**
(C) 채택. 관리자 비밀번호는 다시 평문 환경변수 `ADMIN_PASSWORD`로 저장하고, 로그인 시 `crypto.timingSafeEqual`로 비교한다(D-11 이전 방식으로 복귀).

**결정 사유**
해시 저장이 막으려던 위험은 "Vercel 환경변수 설정 화면에 접근 권한이 있는 사람이 비밀번호 원문을 보는 것"인데, 이 프로젝트는 그 화면에 접근할 수 있는 사람이 사실상 관리자 본인뿐이라 실질적 위험이 낮다. 반면 스크립트를 실행해 해시를 생성하는 절차는 매번 겪어야 하는 실질적 불편이다. `phoneHash`/`phoneEncrypted`(D-10)는 DB(Turso)에 대량으로 쌓이는 회원 개인정보를 외부 서비스로부터 보호하는 것이라 위험 대비 효과가 뚜렷했지만, 관리자 비밀번호 1개를 위해 같은 수준의 장치를 두는 것은 이 프로젝트 규모에서 과설계로 판단했다(단순한 아키텍처 우선 원칙에 부합). `lib/security/password.ts`는 더 이상 두지 않는다.

## D-14. 관리자 대기 승인/수동 예약 추가 — 슬롯 초과 강제 승인 허용 (§10 개정)

**배경**
원래 요구사항(10번)은 "슬롯 초과 강제 승인은 지원하지 않음"으로, 관리자가 대기 예약을 승인하려 해도 슬롯 여유가 없으면 거부했다(Phase 3 구현 및 QA에서 그대로 확인됨). 그런데 실제 운영 관점에서 재검토한 결과, 관리자가 예외적으로 정원 이상의 인원을 받아야 하는 상황(예: 지인 부탁, 현장 사정)에서 매번 예약일 수정 화면으로 이동해 슬롯을 먼저 늘린 뒤 다시 승인하러 돌아와야 하는 것이 불필요한 2단계 작업이라는 지적이 있었다.

**검토한 대안**
- (A) 기존 유지 — 슬롯 여유 없으면 승인/수동추가 거부, 관리자가 먼저 예약일 슬롯을 수동으로 늘려야 함
- (B) 관리자 액션(대기 승인, 수동 예약 추가)에 한해 슬롯 부족 시 자동으로 슬롯을 늘리면서 확정 처리. 사용자가 직접 신청하는 공개 폼은 기존과 동일하게 슬롯 부족 시 `WAITING` 유지
- (C) 사용자 셀프 신청에도 동일하게 자동 슬롯 확장 적용(사실상 슬롯 한도 자체가 무의미해짐)

**최종 결정**
(B) 채택. 관리자가 `WAITING` 예약을 승인하거나 수동으로 예약을 추가할 때 슬롯이 부족하면, 예약일의 슬롯 수(분리 슬롯 모드에서는 해당 예약의 `memberType`에 해당하는 세부 슬롯과 `totalSlots`를 함께)를 자동으로 1 늘리고 `CONFIRMED`로 처리한다. 사용자가 직접 신청하는 공개 예약 폼에는 이 자동 확장을 적용하지 않는다(기존과 동일하게 슬롯이 없으면 `WAITING`).

**결정 사유**
관리자가 명시적으로 "이 사람을 확정시키겠다"고 판단해 승인/추가 버튼을 누르는 행위 자체가 이미 정원 조정에 대한 의사결정이므로, 별도로 예약일 수정 화면에서 슬롯을 먼저 늘리게 하는 것은 불필요한 마찰이다. 반면 사용자 셀프 신청까지 자동 확장을 적용하면 관리자가 설정한 정원이 사실상 아무 의미가 없어지고(C안), 슬롯 정책(SEPARATED/COMBINED)의 존재 이유 자체가 흔들린다. 관리자 액션에만 한정함으로써 "슬롯은 사용자에게는 실제 한도, 관리자에게는 최종 재량"이라는 역할 구분이 명확해진다. 수동 예약 추가 시에도 `determineMemberType` 판정은 동일하게 수행하며(자동 확장은 슬롯 한도 검사 단계만 건너뛴다), 중복/재예약 판정 로직도 그대로 적용된다.

## D-15. 예약일 수정 — 슬롯 수를 확정 인원 미만으로 저장하는 것을 차단 (§13 개정)

**배경**
원래 요구사항(13번)은 "슬롯 수를 줄여도 기존 CONFIRMED 예약을 자동으로 WAITING으로 내리지 않고, 대신 경고만 표시한다"였다(Phase 1/3 구현 및 QA에서 그대로 확인됨). 그런데 실제 화면 QA 과정에서, 관리자가 확정 인원보다 적은 슬롯 수를 실수로 입력해도 경고 문구만 뜨고 저장은 그대로 되는 것이 의도한 동작이 아니라는 피드백이 나왔다.

**검토한 대안**
- (A) 기존 유지 — 저장은 항상 허용, 확정 인원 초과 시 경고 문구만 표시
- (B) 저장 시점에 검증해, 새 슬롯 수(분리 슬롯이면 `annualSlots`/`casualSlots` 각각)가 현재 확정 인원보다 작으면 저장 자체를 거부(`ValidationError`)
- (C) 저장은 허용하되, 초과분 확정 예약을 자동으로 `WAITING`으로 내림(슬롯 초과 상태 자체를 시스템이 즉시 해소)

**최종 결정**
(B) 채택. `updateBookingDay`는 새 슬롯 수가 현재 `CONFIRMED` 인원(분리 슬롯이면 풀별로)보다 작으면 저장을 거부하고 검증 오류를 반환한다. 슬롯 수는 항상 "현재 확정 인원 이상"으로만 설정할 수 있다.

**결정 사유**
(A)의 "경고만 표시"는 사용자가 경고를 무시하고 그대로 저장할 수 있어 사실상 안전장치가 아니었다. (C)는 관리자가 슬롯을 잠깐 잘못 입력했다가 되돌리는 사이에 실제 확정 예약자가 대기로 밀려나는 부작용이 있어, "확정 예약은 관리자가 명시적으로 취소하기 전까지 유지된다"는 기존 원칙(D-14의 전제와도 일치)을 깨뜨린다. 저장 시점에 막는 (B)안은 확정 인원을 초과하는 슬롯 상태 자체가 애초에 발생하지 않도록 하므로, 기존에 있던 "슬롯 초과 경고 배너"는 이 변경 이후 도달 불가능한 코드가 되어 함께 제거한다.

## D-16. 관리자 대시보드 경고 지표 — "슬롯 초과 경고" 대신 "대기 인원 발생 경고" (§16 개정)

**배경**
원래 요구사항(16번)은 관리자 대시보드 표시 정보에 "슬롯 초과 경고"(확정 인원이 슬롯 수를 초과한 예약일 표시)를 포함했다. 그런데 D-14(관리자 액션 시 슬롯 부족하면 자동 확장)와 D-15(슬롯 감소 시 확정 인원 미만으로 저장 차단)가 확정된 뒤, 앱의 정상적인 동작 경로로는 "확정 인원 > 슬롯 수" 상태 자체가 더 이상 발생할 수 없다(D-15에서 예약일 상세 화면의 슬롯 초과 경고 배너도 이미 도달 불가능한 코드로 판단해 제거했다). Phase 5(관리자 대시보드) 착수 전 이 사실이 다시 확인되면서, 대시보드에 같은 지표를 넣는 게 의미가 있는지 재검토가 필요해졌다. 유일하게 남는 발생 경로는 Prisma Studio/직접 SQL을 통한 DB 수동 편집뿐이다.

**검토한 대안**
- (A) 기존 유지 — "슬롯 초과 경고"를 대시보드에 그대로 구현. 정상 운영에서는 항상 표시되지 않지만, 수동 DB 편집으로 인한 데이터 무결성 문제를 잡아내는 방어적 진단 용도로 남겨둔다.
- (B) 요구사항에서 완전히 제거 — 대시보드에 관련 경고 지표를 넣지 않는다.
- (C) "슬롯 초과 경고"를 "대기 인원 발생 경고"로 대체 — 확정 인원이 슬롯을 초과한 예약일 대신, `WAITING` 상태 예약이 하나라도 있는 예약일을 강조 표시한다.

**최종 결정**
(C) 채택. 대시보드의 경고 지표를 "확정 인원 > 슬롯 수"에서 "대기 인원(WAITING) 존재 여부"로 바꾼다. `WAITING` 예약이 있다는 것은 그 시점에 해당 풀(SEPARATED면 memberType별 풀, COMBINED면 전체)이 가득 차 있었다는 뜻이다 — 슬롯이 늘어나면 대기자는 `promoteWaitingBookings`로 즉시 자동 승격되므로, 대기자가 남아있다는 것은 지금도 그 풀에 여유가 없다는 신호다.

**결정 사유**
D-14/D-15로 "슬롯 초과" 상태가 정상 운영에서 발생 불가능해졌으므로, 그 지표를 대시보드 핵심 경고로 유지하는 것은 실효성이 없다(평소엔 절대 표시되지 않는 죽은 지표). 반면 "대기 인원 발생"은 실제 운영에서 흔히 발생하고, "슬롯을 늘려야 하나?"를 관리자가 판단하는 데 직접 도움이 되는 실용적 신호다. 수동 DB 편집으로 인한 데이터 무결성 이상 감지는 이번 범위의 요구사항이 아니며, 필요하다면 별도의 운영 점검 스크립트로 다룰 문제로 남겨둔다(MVP 단순성 원칙에 부합).
