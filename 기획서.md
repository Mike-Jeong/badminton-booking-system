# 배드민턴 예약 관리 시스템 — 기획서

기준 문서: `requirements.md` (최종 통합본)
작성 관점: 기획자(PM) — 비즈니스 로직 및 사용자 경험 중심. 기술 구현 방식은 다루지 않음.
목적: 아키텍트/개발자에게 인계할 기능 단위(에픽) 정리, 사용자 플로우 서술, QA 검증용 완료 기준, 기획 관점 이슈, MVP 범위 재확인.

---

## 1. 기능 단위(에픽) 정리

### 에픽 A. 관리자 인증

**개요**: 별도 계정 체계 없이 환경변수 비밀번호로 관리자 페이지에 로그인한다.

**핵심 사용자 플로우**
1. 관리자가 로그인 페이지에서 비밀번호를 입력한다.
2. 입력값이 `ADMIN_PASSWORD`와 일치하면 서명된 HttpOnly 쿠키 기반 세션이 발급되고 관리자 페이지로 이동한다.
3. 비밀번호가 틀리면 로그인 실패 메시지를 보고 다시 시도한다.
4. 세션은 24시간 후 만료되며, 만료 시 다시 로그인해야 한다.
5. 관리자는 로그아웃을 통해 세션을 종료할 수 있다.

**완료 기준(Acceptance Criteria)**
- [ ] 올바른 비밀번호 입력 시 로그인에 성공하고 관리자 페이지 전체에 접근 가능하다.
- [ ] 틀린 비밀번호 입력 시 로그인에 실패하고 명확한 오류 메시지가 노출된다.
- [ ] 로그인하지 않은 상태로 관리자 페이지 URL에 직접 접근하면 로그인 페이지로 리다이렉트된다.
- [ ] 세션 발급 후 24시간이 지나면 세션이 만료되어 재로그인이 요구된다.
- [ ] 로그아웃 시 세션이 즉시 무효화된다.
- [ ] 일반 사용자 화면에는 관리자 로그인 여부가 노출되지 않는다.

### 에픽 B. 예약일 관리

**개요**: 관리자가 예약 가능한 날짜(세션 단위)를 생성/수정/삭제한다. 같은 날짜에 여러 세션(오전/오후 등)을 만들 수 있다.

**핵심 사용자 플로우**
1. 관리자가 날짜, 장소, 듀티 담당자, 전체 슬롯 수, 슬롯 정책(분리/통합), 필요시 세션 라벨을 입력해 예약일을 생성한다.
2. 요일은 입력 항목이 아니며 날짜로부터 자동 계산되어 화면에 표시된다.
3. 분리 슬롯을 선택한 경우 연 멤버 슬롯 수 + 캐주얼 슬롯 수 = 전체 슬롯 수가 맞는지 저장 시점에 검증되고, 맞지 않으면 저장이 거부된다.
4. 예약일 생성이 완료되면 해당 연도/월/요일과 일치하는 월 멤버가 자동으로 예약에 배정된다 (에픽 E 참고).
5. 관리자는 이후에도 예약일 정보(슬롯 수, 장소, 담당자, 공개 여부 등)를 수정할 수 있다.
6. 관리자가 슬롯 수를 늘리면 대기자가 신청 순서대로 자동 승격된다(에픽 H).
7. 관리자가 슬롯 수를 줄이면 기존 확정 인원은 그대로 두되, 확정 인원이 슬롯보다 많다는 경고가 표시된다.
8. 사용자는 공개된(`isOpen`) 예약일만 조회하고 신청할 수 있다.

**완료 기준(Acceptance Criteria)**
- [ ] 같은 날짜로 여러 개의 예약일(세션)을 라벨을 다르게 하여 생성할 수 있다.
- [ ] 요일 입력 필드가 화면에 없고, 날짜 선택 시 요일이 자동으로 표시된다.
- [ ] 분리 슬롯 모드에서 연 슬롯 + 캐주얼 슬롯 합이 전체 슬롯과 다르면 저장이 실패하고 안내 메시지가 뜬다.
- [ ] 통합 슬롯 모드에서는 연/캐주얼 슬롯 입력 없이 전체 슬롯만으로 생성 가능하다.
- [ ] 예약일 생성 직후 조건에 맞는 월 멤버가 자동으로 배정된 것을 예약자 목록에서 확인할 수 있다.
- [ ] 슬롯 수 증가 시 대기자가 순서대로 자동 확정되고, 그 결과(승격 인원 수)가 관리자에게 안내된다.
- [ ] 슬롯 수 감소 시 확정 인원이 강제로 변경되지 않으며, 확정 인원 > 슬롯 수인 경우 경고 문구가 표시된다.
- [ ] `isOpen=false`인 예약일은 사용자 공개 화면에 노출되지 않는다.
- [ ] 관리자는 예약일을 삭제할 수 있고, 삭제 전 연결된 예약 존재 여부를 확인할 수 있는 안내를 받는다.

### 에픽 C. 연 멤버 관리

**개요**: 이름+전화번호 조합으로 식별되는 연 단위 회원을 관리자가 등록/수정/비활성화한다(하드 삭제 미지원, 확정). 사용자는 예약 시 본인이 연 멤버인지 직접 선택하지 않으며, 서버가 이름+전화번호 일치 여부로 판별한다.

**핵심 사용자 플로우**
1. 관리자가 연 멤버 이름, 전화번호, 메모를 입력해 등록한다.
2. 이름+전화번호(정규화 기준) 조합이 이미 등록되어 있으면 중복 등록이 거부된다.
3. 관리자는 연 멤버를 비활성화하여 자동 배정 및 연 멤버 판별 대상에서 제외할 수 있다.
4. 사용자가 예약 신청 시 입력한 이름과 전화번호가 모두 일치하는 활성 연 멤버가 있으면 해당 예약은 `ANNUAL`로 분류되고, 하나라도 다르면 `CASUAL`로 분류된다.

**완료 기준(Acceptance Criteria)**
- [ ] 이름+전화번호(정규화 후 기준)가 동일한 연 멤버를 중복 등록하면 오류가 발생한다.
- [ ] 이름만 같고 전화번호가 다르면 별개 레코드로 등록 가능하다(오탐 방지 확인).
- [ ] 이름과 전화번호가 모두 일치하는 사용자가 예약하면 `memberType=ANNUAL`로 처리된다.
- [ ] 이름만 일치하거나 전화번호만 일치하는 사용자는 `memberType=CASUAL`로 처리된다.
- [ ] 연 멤버를 비활성화하면 이후 신규 예약 판별 시 캐주얼로 처리된다.
- [ ] 사용자 화면 어디에도 연 멤버 여부를 직접 선택하는 UI가 없다.
- [ ] 관리자 화면에서 연 멤버 목록을 이름/활성여부 등으로 조회할 수 있다.

### 에픽 D. 월 멤버 관리

**개요**: 특정 연도·월·요일에 우선 배정되는 멤버를 연 멤버 중에서 지정한다.

**핵심 사용자 플로우**
1. 관리자가 연 멤버, 적용 연도, 적용 월, 적용 요일을 선택해 월 멤버를 등록한다.
2. 같은 연 멤버를 같은 달 여러 요일(예: 화/목)로 각각 등록할 수 있다.
3. 동일한 연 멤버+연도+월+요일 조합을 다시 등록하려 하면 거부된다.
4. 관리자는 월 멤버를 수정하거나 비활성화할 수 있다(하드 삭제 미지원, 확정).

**완료 기준(Acceptance Criteria)**
- [ ] 김민수를 2026년 7월 화요일, 2026년 7월 목요일 두 건으로 각각 등록할 수 있다.
- [ ] 김민수를 2026년 7월 화요일로 중복 등록하면 거부된다.
- [ ] 월 멤버 등록 화면에서 대상은 반드시 기존 연 멤버 중에서 선택하며, 신규 이름을 직접 입력해 등록할 수 없다.
- [ ] 월 멤버를 비활성화하면 이후 자동 배정 대상에서 제외된다.
- [ ] 관리자는 연도/월 단위로 월 멤버 목록을 필터링해 조회할 수 있다.

---

### 에픽 E. 월 멤버 자동 배정

**개요**: 예약일의 연도/월/요일과 일치하는 활성 월 멤버를 자동으로 예약에 추가한다.

**핵심 사용자 플로우**
1. 관리자가 예약일을 새로 생성하면, 해당 연도/월/요일에 맞는 활성 월 멤버가 자동으로 조회된다.
2. 연결된 연 멤버가 비활성 상태이면 월 멤버가 활성이어도 자동 배정에서 제외된다.
3. 조건에 맞는 월 멤버마다 예약이 자동 생성되며, `memberType=ANNUAL`, `source=MONTHLY_MEMBER_AUTO`로 기록된다.
4. 분리 슬롯 모드에서는 연 멤버 슬롯을 소비하며, 슬롯이 있으면 `CONFIRMED`, 없으면 `WAITING`으로 생성된다.
5. 같은 예약일에 이미 `WAITING`/`CONFIRMED` 상태의 동일 이름+전화번호 예약이 있으면 새로 만들지 않는다.
6. 같은 예약일에 `CANCELLED` 이력만 있는 경우에도 소스와 관계없이 새 예약을 재생성한다(확정 사항 변경: 더 이상 스킵하지 않음). 특정 멤버를 앞으로 계속 배정에서 제외하려면 취소 이력이 아니라 해당 월 멤버 등록 자체를 비활성화해서 처리한다.
7. 관리자는 예약일 상세 화면의 "월 멤버 자동 배정" 버튼, 또는 월 멤버 관리 화면의 "해당 월 예약일에 자동 배정" 버튼으로 동일 로직을 재실행할 수 있다.
8. 자동 배정 결과로 생성 건수와 스킵 건수가 관리자에게 안내된다.

**완료 기준(Acceptance Criteria)**
- [ ] 예약일(2026-07-07, 화)을 생성하면 2026년 7월/화요일 월 멤버가 자동으로 예약에 추가된다.
- [ ] 월 멤버는 활성인데 연결된 연 멤버가 비활성인 경우 자동 배정되지 않는다.
- [ ] 슬롯이 부족한 상태에서 자동 배정되는 월 멤버는 `WAITING`으로 생성된다.
- [ ] 동일 예약일에 이미 확정/대기 중인 동일인 예약이 있으면 자동 배정이 중복 생성하지 않는다.
- [ ] 동일 예약일에 취소 이력만 있는 동일인도 자동 배정 시 새 예약이 재생성된다(소스 무관, 스킵하지 않음).
- [ ] 예약일 상세 화면의 "월 멤버 자동 배정" 버튼 실행 시 이미 배정된 인원은 재생성되지 않는다(재실행 안전성).
- [ ] 월 멤버 관리 화면에서 특정 연/월(및 선택적으로 요일)에 대해 일괄 자동 배정을 실행하면 해당 월의 모든 예약일에 반영된다.
- [ ] 자동 배정 실행 후 생성 건수/스킵 건수가 관리자 화면에 표시된다.

### 에픽 F. 사용자 예약 신청

**개요**: 회원가입 없이 이름+전화번호만으로 예약을 신청한다. 서버가 연/캐주얼 여부와 확정/대기 여부를 판별한다.

**핵심 사용자 플로우**
1. 사용자가 공개된 예약일 목록에서 원하는 날짜(세션)를 선택한다.
2. 이름, 전화번호를 입력해 예약을 신청한다.
3. 서버는 이름/전화번호를 정규화하고, 연 멤버 일치 여부로 `ANNUAL`/`CASUAL`을 판별한다.
4. 같은 예약일에 동일 이름+전화번호로 이미 `WAITING` 또는 `CONFIRMED` 예약이 있으면 신청이 거부된다.
5. 같은 예약일에 동일 이름+전화번호의 `CANCELLED` 이력만 있다면 새 예약(재예약)이 허용된다.
6. 슬롯 정책과 잔여 슬롯을 확인해 여유가 있으면 `CONFIRMED`, 없으면 `WAITING`으로 예약이 생성된다.
7. 신청 완료 후 사용자는 본인의 확정/대기 상태를 안내받는다.
8. 사용자는 같은 예약일의 다른 예약자 이름(및 확정/대기 상태)을 볼 수 있으나 전화번호는 볼 수 없다.

**완료 기준(Acceptance Criteria)**
- [ ] 이름/전화번호 외의 추가 정보 입력 없이 예약 신청이 완료된다.
- [ ] 사용자가 `memberType`을 직접 선택하는 입력 요소가 존재하지 않는다.
- [ ] 연 멤버로 등록된 이름+전화번호로 신청하면 자동으로 연 멤버 슬롯(분리 모드) 또는 전체 슬롯(통합 모드) 기준으로 판단된다.
- [ ] 동일 예약일에 이미 확정/대기 중인 동일인이 재신청하면 "이미 예약됨" 안내와 함께 거부된다.
- [ ] 동일 예약일에 취소 이력만 있는 동일인은 재신청(새 예약 생성)이 가능하다.
- [ ] 슬롯이 가득 찬 상태에서 신청하면 `WAITING`으로 등록되고 대기 순번 관련 안내를 받는다(예: "현재 대기 중입니다").
- [ ] 신청 직후 화면에서 본인 예약이 확정인지 대기인지 명확히 구분되어 표시된다.
- [ ] 공개 화면에서 다른 예약자의 이름과 상태(확정/대기)는 보이지만 전화번호는 어디에도 노출되지 않는다.
- [ ] `isOpen=false`인 예약일에는 신청 자체가 불가능하다.

### 에픽 G. 예약 취소 (사용자)

**개요**: 사용자는 전화번호로 본인 예약을 조회한 뒤, 특정 예약을 선택해 취소한다(2단계 플로우).

**핵심 사용자 플로우**
1. 사용자가 전화번호를 입력해 본인 예약 목록(날짜, 이름, 상태)을 조회한다.
2. 목록에서 취소할 예약(bookingId)을 선택한다.
3. 서버는 선택한 bookingId의 등록된 전화번호와 사용자가 입력한 전화번호(정규화 후)가 일치하는지 재확인한다.
4. 일치하면 해당 예약 상태를 `CANCELLED`로 변경하고 취소 시각을 기록한다(레코드는 삭제되지 않음).
5. 취소된 예약이 `CONFIRMED` 상태였다면 대기자 중 다음 순번이 자동으로 `CONFIRMED`로 승격된다.
6. 사용자는 취소 완료 안내를 받는다.
7. 사용자는 취소 후에도 원하면 같은 예약일에 본인 명의로 다시 예약을 신청할 수 있다.

**완료 기준(Acceptance Criteria)**
- [ ] 전화번호만 입력하면 해당 번호로 등록된 모든 예약(여러 날짜/여러 이름 포함)이 목록으로 조회된다.
- [ ] 목록에서 bookingId 단위로 개별 취소를 선택할 수 있다.
- [ ] 취소 요청 시 입력한 전화번호가 해당 bookingId의 전화번호와 다르면 취소가 거부된다.
- [ ] 취소된 예약은 DB에서 삭제되지 않고 상태만 `CANCELLED`로 변경되며 취소 시각이 기록된다.
- [ ] `CONFIRMED` 예약을 취소하면 대기 1순위가 자동으로 `CONFIRMED`로 승격된다.
- [ ] `WAITING` 예약을 취소해도 다른 대기자의 순번/상태에는 영향이 없다.
- [ ] 취소 후 같은 사용자가 같은 예약일에 다시 신청하면 새로운 예약 건으로 정상 생성된다.

### 에픽 H. 대기열 관리 및 자동 승격 로직

**개요**: 슬롯 여유에 따라 확정/대기를 자동 판별하고, 슬롯 증가·취소 발생 시 대기자를 규칙에 따라 승격한다. 관리자는 대기 예약을 개별 승인할 수도 있다.

**핵심 사용자 플로우**
1. 예약 신청 시점에 슬롯 여유가 있으면 자동 `CONFIRMED`, 없으면 `WAITING`으로 처리된다(분리 슬롯은 연/캐주얼 각각 별도 판단).
2. 관리자가 슬롯 수를 늘리면 대기자를 신청 순서(동시간이면 id 낮은 순)로 늘어난 만큼 자동 확정한다.
3. `CONFIRMED` 예약이 취소되면 대기 1순위가 자동으로 확정된다.
4. 관리자는 대기 중인 개별 예약을 슬롯 여유가 있을 때 수동으로 승인할 수 있으며, 이때는 FIFO 순서를 강제하지 않는다.
5. 분리 슬롯 정책에서는 연 슬롯과 캐주얼 슬롯 간 교차 승격이 발생하지 않는다(연 슬롯이 남아도 캐주얼 대기자는 승격되지 않음).

**완료 기준(Acceptance Criteria)**
- [ ] 슬롯이 가득 찬 상태의 신규 신청은 항상 `WAITING`으로 생성된다.
- [ ] 슬롯 20명 확정, 대기 3명 상태에서 슬롯을 22명으로 늘리면 대기 2명이 신청 순서대로 자동 확정된다.
- [ ] 슬롯 증가로 승격될 인원이 여러 명일 때, 먼저 신청한 사람이 우선 승격된다. 신청 시각이 같으면 id가 작은 예약이 우선한다.
- [ ] `CONFIRMED` 취소 발생 시 대기 1순위 1명만 자동 확정되고 나머지 대기자는 대기 상태를 유지한다.
- [ ] 관리자가 대기 예약을 개별 승인할 때, 슬롯 여유가 없으면 승인이 거부된다.
- [ ] 관리자의 개별 승인은 대기 순번을 건너뛰어도(FIFO 미준수) 허용된다.
- [ ] 분리 슬롯 모드에서 연 슬롯에 여유가 생겨도 캐주얼 대기자는 승격되지 않고, 반대의 경우도 동일하다.
- [ ] 통합 슬롯 모드에서는 연/캐주얼 구분 없이 전체 대기열 순서로 승격된다.

### 에픽 I. 관리자 예약 운영

**개요**: 관리자가 특정 예약일의 예약자 목록을 확인하고 상태를 관리(승인/취소/수정)한다.

**핵심 사용자 플로우**
1. 관리자가 예약일을 선택하면 해당 날짜의 전체 예약자 목록(이름, 전화번호, 상태, 유형, source)을 확인한다.
2. 관리자는 특정 예약의 상태를 변경(대기→확정 등)하거나 취소 처리할 수 있다.
3. 관리자가 예약을 취소 처리하면 사용자 취소와 동일하게 대기자 자동 승격이 발생한다(취소 대상이 `CONFIRMED`였던 경우).
4. 관리자는 필요 시 수동으로 신규 예약을 추가할 수 있다(source=ADMIN).

**완료 기준(Acceptance Criteria)**
- [ ] 관리자 화면에서 예약일별 예약자 목록에 이름/전화번호/상태/유형(ANNUAL·CASUAL)/source가 모두 표시된다.
- [ ] 관리자가 대기 예약을 확정으로 변경하면 슬롯 여유 검사가 수행된다.
- [ ] 관리자가 예약을 취소 처리하면 `cancelledAt`이 기록되고 대기자 자동 승격 로직이 동일하게 실행된다.
- [ ] 관리자가 직접 추가한 예약은 `source=ADMIN`으로 기록된다.
- [ ] 관리자가 추가한 예약도 이름+전화번호 기준 중복/재예약 규칙(18번)을 동일하게 적용받는다.

### 에픽 J. 관리자 대시보드

**개요**: 관리자가 전체 예약 현황을 한눈에 파악할 수 있는 요약 화면.

**핵심 사용자 플로우**
1. 관리자가 대시보드에 접속하면 오늘 날짜(Pacific/Auckland 기준) 예약 현황이 먼저 보인다.
2. 날짜별 확정/대기/취소 인원과 슬롯 사용률을 확인할 수 있다.
3. 슬롯이 초과된(확정 인원 > 슬롯) 예약일에는 경고가 표시된다.

**완료 기준(Acceptance Criteria)**
- [ ] 대시보드의 "오늘" 기준은 Pacific/Auckland 타임존 기준으로 계산된다(서버 로컬 시간이나 UTC 기준이 아님).
- [ ] 날짜별로 확정/대기/취소 인원 수가 구분되어 표시된다.
- [ ] 슬롯 사용률(확정 인원/전체 슬롯)이 표시된다.
- [ ] 확정 인원이 슬롯 수를 초과한 예약일에는 시각적 경고(예: 색상/아이콘/문구)가 표시된다.
- [ ] 같은 날짜에 여러 세션(라벨)이 있는 경우 세션별로 구분되어 표시된다.

## 2. 기획 관점 이슈 (문서에 없는 신규 시나리오만)

문서의 기존 확정 사항을 뒤집는 제안이 아니라, 문서에 아예 언급이 없는 상황에 대한 질문/제안입니다. 개발 착수 전 기획 차원에서 확인이 필요합니다.

1. ~~**연/월 멤버 "삭제"의 의미**~~ — **해결됨**: 하드 삭제는 지원하지 않고 `isActive=false`(비활성화)로 통일하기로 확정. `decisions.md` D-07 참고. requirements.md 4·5·15번, 본 문서 에픽 C·D에 반영됨.
2. **지난 예약일(과거 날짜)의 취급**: 날짜가 지난 예약일에 대해 사용자가 여전히 신규 신청/취소를 시도할 수 있는지, 관리자 화면에서 과거 예약일을 별도로 구분해 보여줄지에 대한 언급이 없습니다. 운영상 "지난 예약일은 자동으로 신청 마감" 같은 정책이 필요한지 확인 필요.
3. **동일 전화번호로 다른 이름을 반복 입력하는 어뷰징**: 18번 중복 방지는 "이름+전화번호" 조합 기준이라, 같은 사람이 이름을 살짝 바꿔가며 여러 자리를 예약하는 것을 막지 못합니다. 동호회 특성상 큰 문제가 아닐 수 있으나, 이런 상황이 실제 발생했을 때 관리자가 어떻게 인지/조치할지(예: 전화번호 기준 예약 목록에서 쉽게 식별 가능하게) 화면 설계 시 고려가 필요합니다.
4. **예약 신청/취소에 대한 알림 수단 부재**: 별도 회원가입이 없으므로 이메일/SMS 등 알림 발송 여부가 문서에 언급되어 있지 않습니다. 신청 직후 화면 안내 외에, 대기→확정 승격처럼 사용자가 그 자리에 없을 때 발생하는 상태 변화를 사용자가 알 방법이 없습니다(다시 조회해야만 확인 가능). MVP 이후 SMS 알림 등의 필요성을 별도로 논의할 가치가 있습니다.
5. **취소 확인(되돌리기 방지) UX**: 취소는 되돌릴 수 없는 동작(재예약은 가능하지만 슬롯 상황에 따라 대기로 밀릴 수 있음)인데, 실수 취소를 막기 위한 확인 절차(예: "정말 취소하시겠습니까?")에 대한 언급이 없습니다.
6. **전화번호 형식 검증 범위**: 정규화 규칙(7번)은 "숫자만 남김"까지만 정의되어 있고, 최소/최대 자릿수 검증이나 빈 문자열 처리에 대한 언급이 없습니다. 잘못된 전화번호(예: 3자리)로 예약이 생성되는 것을 막을지 여부가 기획적으로 결정되어야 합니다.
7. **같은 날짜 다중 세션(라벨) 상황에서 사용자 조회 UX**: 같은 날짜에 "오전/오후" 세션이 있을 때 사용자에게 어떻게 구분해 보여줄지(달력 뷰, 리스트 뷰 등)에 대한 화면 설계 방향이 문서에 없어 화면 기획 단계에서 별도로 정의가 필요합니다.

## 3. MVP 범위 재확인 (22번 개발 우선순위 기준)

문서 22번은 개발 착수 순서이자 사실상 MVP 항목 목록입니다. 기획자 관점에서 각 항목의 "필수 여부"를 재확인합니다.

| # | 항목 | 구분 | 비고 |
|---|------|------|------|
| 1 | 관리자 로그인 | 필수(MVP) | 관리자 기능 전체의 전제조건. |
| 2 | 예약일 생성 | 필수(MVP) | 서비스 성립의 최소 조건. |
| 3 | 사용자 예약 신청 | 필수(MVP) | 서비스의 핵심 가치. |
| 4 | 자동 승인/대기 처리 | 필수(MVP) | 슬롯 기반 확정/대기 판별 없이는 예약 신청이 의미가 없음. |
| 5 | 예약 취소 | 필수(MVP) | 취소 없는 예약 시스템은 실사용이 불가능. |
| 6 | 관리자 예약 관리 | 필수(MVP) | 운영자가 현장 문제(노쇼, 오입력 등)를 처리할 최소 수단. |
| 7 | 연 멤버 관리 | 필수(MVP) | 월 멤버 자동배정(핵심 요구사항)의 전제조건. |
| 8 | 월 멤버 관리 | 필수(MVP) | 동호회 운영 방식상 최초 요구사항에 명시된 핵심 기능. |
| 9 | 월 멤버 자동 배정 | 필수(MVP) | 이 시스템을 만드는 핵심 동기(수기 관리 대체)에 해당. |
| 10 | 슬롯 분리/통합 정책 | 필수(MVP) | 예약일 생성 시점부터 필요한 기본 옵션이라 별도 분리 불가. |
| 11 | 관리자 대시보드 | 필수(MVP, 단 경량화 가능) | "오늘 현황 + 슬롯 초과 경고" 등 핵심 지표만 1차로 구현하고, 상세 통계/차트 등은 차후 개선으로 미뤄도 무방. |
| 12 | UI/UX 개선 | 차후 개선 | 기능적으로는 1~11번이 동작하면 서비스 성립. 디자인 다듬기, 반응형 최적화, 인터랙션 개선 등은 MVP 출시 이후 반복 개선 대상. |

**종합 의견**: 문서 자체가 "22번 우선순위 = MVP 개발 순서"로 설계되어 있어, 1~11번은 사실상 모두 MVP 필수 범위로 보는 것이 타당합니다(어느 하나라도 빠지면 핵심 유스케이스—연/월 멤버 우선 배정—가 성립하지 않음). 유일하게 성격이 다른 것은 12번(UI/UX 개선)으로, 이는 "완료 기준이 없는 지속 개선 항목"이라 별도 스프린트로 분리하는 것을 권장합니다. 11번(대시보드)은 필수이되 최초 릴리즈에서는 핵심 지표만 담은 경량 버전으로 범위를 좁히는 것을 제안합니다.
